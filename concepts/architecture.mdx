## Overview

PardoX is built on a unique architectural foundation that combines **Rust's performance and safety** with **universal SDK accessibility**. The architecture is designed around three core pillars:

1. **High-performance Rust core** — All data processing happens in native code
2. **Zero-copy memory design** — Data flows without intermediate copies
3. **Multi-language SDK layer** — Python, JavaScript, and PHP interfaces to a single engine

---

## The Rust Core

### HyperBlockManager: Columnar Storage Engine

At the heart of PardoX is the **HyperBlockManager**, a columnar storage engine written entirely in Rust. It manages data in contiguous memory blocks optimized for CPU cache efficiency.

```rust
// Conceptual representation
struct HyperBlockManager {
    blocks: Vec<HyperBlock>,
    schema: Schema,
    row_count: usize,
}
```

Key characteristics:

- **Columnar layout** — Each column stored as a separate contiguous array
- **Type-safe** — Enforces `Int64`, `Float64`, and `Utf8` types at compile time
- **Memory-mapped I/O** — Large files mapped directly into virtual memory
- **Thread-safe** — Built on Rust's ownership model with zero data races

<Note>
The HyperBlockManager is never exposed directly to SDK languages. All interactions happen through FFI function pointers, ensuring memory safety across language boundaries.
</Note>

### SIMD Acceleration

PardoX leverages **SIMD (Single Instruction, Multiple Data)** CPU instructions for vectorized operations:

| Platform | Instruction Set | Speedup |
|----------|----------------|----------|
| Intel/AMD x64 | AVX2 | 5x-20x |
| Apple Silicon (M1/M2/M3) | NEON | 8x-25x |
| ARM Linux | NEON | 6x-18x |

**Example: Vectorized multiplication**

```python
# Python SDK call
df['revenue'] = df['price'] * df['quantity']
```

```rust
// Rust execution (simplified)
fn simd_multiply_f64(a: &[f64], b: &[f64]) -> Vec<f64> {
    a.iter()
        .zip(b.iter())
        .map(|(x, y)| x * y)  // Auto-vectorized by LLVM to AVX2/NEON
        .collect()
}
```

<Tip>
SIMD operations process 4-8 floating-point values per CPU cycle, compared to Python's interpreted loops which process one value per dozens of cycles.
</Tip>

## Zero-Copy Architecture

### Memory Flow: CSV to DataFrame

PardoX eliminates intermediate copies throughout the data pipeline:

<Steps>
  <Step title="Memory-mapped file">
    CSV file is `mmap`-ed into virtual memory — no `read()` syscalls
  </Step>
  
  <Step title="Parallel parsing">
    File split into chunks, parsed by worker threads directly into Rust buffers
  </Step>
  
  <Step title="Columnar consolidation">
    Parsed values written to HyperBlock arrays — no Python objects created
  </Step>
  
  <Step title="SDK pointer return">
    Opaque pointer to HyperBlockManager returned to Python/JS/PHP
  </Step>
</Steps>

**Traditional approach (Pandas):**
```
Disk → OS Buffer → Python Strings → Pandas Objects → NumPy Arrays
          ↓              ↓               ↓
        Copy 1        Copy 2         Copy 3
```

**PardoX approach:**
```
Disk → mmap → Rust Buffers → SDK Pointer
                  ↓
             Zero Copies
```

### FFI Bridge Design

The Foreign Function Interface (FFI) layer exposes Rust functions as C-compatible ABI:

```rust
// Rust FFI export
#[no_mangle]
pub extern "C" fn pardox_load_manager_csv(
    path: *const c_char,
    schema_json: *const c_char,
    config_json: *const c_char,
) -> *mut HyperBlockManager {
    // Implementation omitted
}
```

```python
# Python SDK binding (via ctypes)
lib = ctypes.CDLL("pardox-cpu-Linux-x64.so")
lib.pardox_load_manager_csv.argtypes = [c_char_p, c_char_p, c_char_p]
lib.pardox_load_manager_csv.restype = c_void_p

# User call
df = px.read_csv("data.csv")
# df._ptr holds the raw C pointer to HyperBlockManager
```

<Warning>
SDK users never access `_ptr` directly. All operations go through safe Python/JS/PHP methods that validate inputs before calling Rust.
</Warning>

## Multi-SDK Approach

### Single Rust Core, Three SDK Layers

PardoX compiles **one Rust library** into platform-specific binaries:

```
pardox-cpu-Windows-x64.dll      (Windows)
pardox-cpu-Linux-x64.so         (Linux)
pardox-cpu-MacOS-ARM64.dylib    (Apple Silicon)
pardox-cpu-MacOS-Intel.dylib    (Intel Mac)
```

Each SDK loads the appropriate binary:

<CodeGroup>
```python Python SDK
import ctypes
import sys

if sys.platform == "linux":
    lib = ctypes.CDLL("pardox-cpu-Linux-x64.so")
elif sys.platform == "darwin":
    if platform.machine() == "arm64":
        lib = ctypes.CDLL("pardox-cpu-MacOS-ARM64.dylib")
```

```javascript JavaScript SDK
const koffi = require('koffi');
const os = require('os');

let libPath;
if (os.platform() === 'linux') {
  libPath = './pardox-cpu-Linux-x64.so';
} else if (os.platform() === 'darwin') {
  libPath = os.arch() === 'arm64' 
    ? './pardox-cpu-MacOS-ARM64.dylib'
    : './pardox-cpu-MacOS-Intel.dylib';
}

const lib = koffi.load(libPath);
```

```php PHP SDK
<?php
$os = PHP_OS_FAMILY;
if ($os === 'Linux') {
    $lib = FFI::cdef(
        "void* pardox_load_manager_csv(...);",
        __DIR__ . "/pardox-cpu-Linux-x64.so"
    );
}
```
</CodeGroup>

### API Parity Across SDKs

All three SDKs expose identical functionality:

| Feature | Python | JavaScript | PHP |
|---------|--------|------------|-----|
| CSV Read | ✅ `px.read_csv()` | ✅ `px.readCsv()` | ✅ `PardoX::readCsv()` |
| DataFrame Operations | ✅ `df['col']` | ✅ `df.col()` | ✅ `$df->col()` |
| Arithmetic | ✅ `df['a'] * df['b']` | ✅ `df.mul('a', 'b')` | ✅ `$df->mul('a', 'b')` |
| Database I/O | ✅ `read_sql()` | ✅ `readSql()` | ✅ `readSql()` |
| GPU Sort | ✅ `df.sort_values(gpu=True)` | ✅ `df.sortValues({gpu: true})` | ✅ `$df->sortValues('col', true)` |
| NumPy Bridge | ✅ `np.array(df['col'])` | ❌ | ❌ |

<Info>
The NumPy zero-copy bridge is Python-specific because it uses the `__array__` protocol. JavaScript and PHP use JSON serialization for interop.
</Info>

## Native Database Integration

### Relational Conqueror: Wire Protocol in Rust

Unlike traditional ORMs, PardoX implements database wire protocols **entirely in Rust**:

<CardGroup cols={2}>
  <Card title="PostgreSQL" icon="database">
    Driver: `tokio-postgres`  
    Protocol: Binary COPY FROM STDIN
  </Card>
  <Card title="MySQL" icon="database">
    Driver: `mysql` crate v25  
    Protocol: LOAD DATA LOCAL INFILE
  </Card>
  <Card title="SQL Server" icon="database">
    Driver: `tiberius` v0.12  
    Protocol: Multi-row INSERT
  </Card>
  <Card title="MongoDB" icon="database">
    Driver: `mongodb` crate v2.8  
    Protocol: Bulk insert_many
  </Card>
</CardGroup>

**Why this matters:**

When you call `df.to_sql()` from **any SDK** (Python, Node.js, or PHP), the execution path is:

```
SDK (Python/JS/PHP) → FFI → Rust Core → Database Driver → TCP Socket
                                ↓
                    No Python/JS/PHP database library involved
```

This means:
- **No psycopg2** in Python
- **No mysql2** in Node.js  
- **No PDO** in PHP

The entire data transfer happens at native Rust speed, regardless of which language called the function.

<Check>
**Performance impact:** PostgreSQL bulk writes achieve **50x speedup** vs. traditional Python `psycopg2` INSERT loops by using `COPY FROM STDIN` protocol.
</Check>

## GPU Acceleration Pipeline

### WebGPU Bitonic Sort

PardoX includes an **optional GPU sort** for large datasets:

<Steps>
  <Step title="GPU availability check">
    Rust checks for `wgpu` backend (Vulkan/Metal/DirectX12)
  </Step>
  
  <Step title="Buffer upload">
    Column data uploaded to GPU memory as compute buffer
  </Step>
  
  <Step title="Bitonic sort shader">
    WGSL shader executes parallel sort on GPU
  </Step>
  
  <Step title="Result download">
    Sorted indices downloaded back to CPU
  </Step>
  
  <Step title="DataFrame reorder">
    Rust reorders all columns using sorted indices
  </Step>
</Steps>

**Automatic fallback:**

If GPU is unavailable, PardoX falls back to CPU parallel merge sort:

```python
# User code
sorted_df = df.sort_values("price", gpu=True)

# Rust logic (pseudocode)
if gpu_available() {
    return gpu_bitonic_sort(column);
} else {
    eprintln!("[PardoX GPU Sort] GPU not available, using CPU sort.");
    return cpu_parallel_sort(column);
}
```

<Tip>
For datasets under 10M rows, CPU sort is often faster due to GPU upload/download overhead. GPU shines at 50M+ rows.
</Tip>

## Binary Format: `.prdx`

### Native Persistence Format

The `.prdx` format is a custom binary layout designed for instant loading:

```
[Header: 256 bytes]
  - Magic number: 0x50 0x52 0x44 0x58 ("PRDX")
  - Version: u32
  - Row count: u64
  - Column count: u32
  - Schema offset: u64

[Schema Block]
  - Column names (null-terminated UTF-8)
  - Column types (u8: 0=Int64, 1=Float64, 2=Utf8)

[Data Blocks]
  - Columnar arrays (aligned to 64-byte cache lines)
  - No compression (optimized for mmap)
```

**Read performance:**

| Format | 2GB File Read Time |
|--------|-------------------|
| CSV (multi-threaded) | ~8 seconds |
| Parquet (Apache Arrow) | ~3 seconds |
| **PRDX (memory-mapped)** | **~0.5 seconds** |

<Info>
PRDX achieves **~4.6 GB/s** throughput on NVMe SSDs because it's essentially a direct memory copy with minimal parsing.
</Info>

## Cross-Platform Compilation

PardoX uses Rust's cross-compilation toolchain to build for all platforms from a single codebase:

```bash
# Linux x64
cargo build --release --target x86_64-unknown-linux-gnu

# Windows x64
cargo build --release --target x86_64-pc-windows-msvc

# macOS Intel
cargo build --release --target x86_64-apple-darwin

# macOS Apple Silicon
cargo build --release --target aarch64-apple-darwin
```

The resulting binaries are packaged into SDK distributions:

```
python_sdk/pardox/libs/
  ├── Linux/pardox-cpu-Linux-x64.so
  ├── Win/pardox-cpu-Windows-x64.dll
  └── Mac/
      ├── pardox-cpu-MacOS-ARM64.dylib
      └── pardox-cpu-MacOS-Intel.dylib

js_sdk/pardox/lib/
  └── (same structure)

php_sdk/pardox/lib/
  └── (same structure)
```

## Memory Safety Guarantees

### Rust Ownership Model

PardoX benefits from Rust's compile-time memory safety:

```rust
// This code won't compile — prevents use-after-free
let df = load_csv("data.csv");
drop(df);  // Memory freed
let col = df.get_column("price");  // ❌ Compile error: `df` already moved
```

### FFI Safety Layer

All FFI functions validate pointers before dereferencing:

```rust
#[no_mangle]
pub extern "C" fn pardox_get_row_count(ptr: *mut HyperBlockManager) -> i64 {
    if ptr.is_null() {
        eprintln!("[ERROR] Null pointer passed to pardox_get_row_count");
        return -1;  // Error code
    }
    let manager = unsafe { &*ptr };
    manager.row_count() as i64
}
```

<Warning>
SDK developers must call `pardox_free_manager()` when done with a DataFrame. Python's `__del__` method handles this automatically.
</Warning>
